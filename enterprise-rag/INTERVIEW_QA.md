# Enterprise RAG Platform Interview Q&A

## 1. 这个项目的定位是什么？

**问题**

这个项目解决什么问题，目标用户是谁？

**回答**

这是一个面向企业私有化部署的 RAG 平台后端基座，目标是把企业知识库问答从原型推进到可落地平台。它不是单一问答 Demo，而是包含多租户、权限、审计、评测、运维观测、异步索引、向量检索和可插拔模型接入的一套平台化框架。

目标用户主要有三类：

- 平台研发团队：需要稳定扩展模型、向量库、任务系统和权限体系。
- 企业内部业务团队：需要上传文档、构建知识库、进行问答。
- 运维和安全团队：需要审计、认证、配置治理和交付能力。

## 2. 为什么这个项目不是直接写一个“问答接口”就结束了？

**问题**

为什么要做这么多平台层能力，比如多租户、审计、评测？

**回答**

企业级 RAG 的难点不在“调用一次大模型”，而在于：

- 数据隔离：不同租户、部门、知识库之间必须隔离。
- 结果可追踪：回答必须能追溯到文档和切片。
- 质量可治理：需要知道召回是否正确、引用是否命中、反馈是否有效。
- 交付可运维：要能部署、扩缩容、监控、告警、审计。

所以平台层能力是一期开工就必须考虑的基线，而不是后补。

## 3. 这个项目的整体架构是什么？

**问题**

请概括一下项目的技术架构和分层。

**回答**

项目大致分成六层：

1. 接入层：FastAPI API、路由、统一异常、请求上下文、中间件。
2. 认证与治理层：OIDC、API Token、RBAC、审计。
3. 应用服务层：租户、知识库、文档、问答、评测、运营、溯源等服务。
4. 能力抽象层：LLM、Embedding、Parser、Reranker、VectorStore 的 Provider 抽象。
5. 数据层：SQLAlchemy ORM、Repository、Alembic。
6. 交付层：Celery、Docker、Compose、K8s、OpenAPI、Postman、联调脚本。

这样做的目的是降低外部依赖耦合，保证模型、向量库、任务系统都能替换。

## 4. 为什么用了 Provider 抽象？

**问题**

为什么要抽象 `LLMProvider`、`EmbeddingProvider`、`VectorStoreProvider` 这些接口？

**回答**

这是为了避免把平台绑死在某个厂商或某个基础设施实现上。

例如：

- LLM 可以是 OpenAI-compatible 网关，也可以是企业内部模型网关。
- Embedding 可以是第三方模型，也可以是本地向量模型。
- VectorStore 可以是内存、`pgvector`，以后也可以接 Milvus。

Provider 抽象带来的收益是：

- 上层业务代码不感知具体厂商。
- 不同环境可以切不同后端。
- 单元测试可以用 stub 或 deterministic provider。

## 5. 现在这个项目已经支持哪些模型接入方式？

**问题**

真实模型接入现在做到什么程度？

**回答**

当前支持的核心模式是 `openai_compatible`，也就是兼容 OpenAI 风格接口的第三方模型网关。

支持的关键配置包括：

- `OPENAI_BASE_URL`
- `OPENAI_API_KEY`
- `OPENAI_CHAT_MODEL`
- `OPENAI_EMBEDDING_MODEL`
- `OPENAI_TIMEOUT_SECONDS`
- `OPENAI_MAX_RETRIES`

目前已经支持：

- 聊天模型调用
- Embedding 模型调用
- Provider 诊断
- Provider 主动探测

也就是说，模型网关只要兼容 `/chat/completions` 和 `/embeddings`，就能接进来。

## 6. 这个项目的检索链路是怎样的？

**问题**

问答时数据是怎么流转的？

**回答**

主流程是：

1. 用户提问。
2. 用 `EmbeddingProvider` 对 query 做向量化。
3. 调 `VectorStoreProvider` 做向量检索。
4. 可选走重排。
5. 拼接 prompt。
6. 调 `LLMProvider` 生成答案。
7. 返回答案和 citations。
8. 记录 query record、retrieval logs、audit event。

这里比较关键的一点是，query 已经不是固定向量，而是真正走 embedding provider。

## 7. 文档入库和索引链路是怎样的？

**问题**

文档上传后，平台如何完成索引构建？

**回答**

流程是异步化的：

1. 上传文档。
2. 创建 `index_job`。
3. 任务调度器分发任务，当前支持 `local` 和 `celery`。
4. Worker 解析文档。
5. 切片并调用 embedding。
6. Chunk 落库。
7. 向量写入向量库。
8. 更新 document 和 index job 状态。

这样做的原因是文档处理天然是重任务，不适合同步阻塞 API 请求。

## 8. 为什么要把 Chunk 落数据库，而不只是放向量库里？

**问题**

向量库里已经有 chunk 内容了，为什么还需要 `chunks` 表？

**回答**

因为企业平台不仅要“能检索”，还要“能治理”。

Chunk 落库的价值包括：

- 支持溯源查询。
- 方便审计和调试。
- 支持离线评测。
- 便于后续做重建、修订、清理。
- 向量库不是所有结构化查询都方便做。

所以数据库和向量库是职责分离的：数据库管治理和可追踪，向量库管相似度检索。

## 9. 为什么选 `pgvector`？

**问题**

为什么先做 `pgvector`，而不是直接上 Milvus？

**回答**

因为 `pgvector` 对当前阶段更实用：

- 交付简单，和 PostgreSQL 一体化。
- 对私有化项目非常友好。
- 运维成本比单独维护一个向量数据库低。
- 对中小规模企业知识库足够。

未来如果数据规模、检索延迟或复杂索引需求上升，再抽象切换到 Milvus 会更合理。

## 10. 这个项目是怎么做认证的？

**问题**

目前支持哪些认证模式？

**回答**

当前支持五种模式：

- `headers`
- `api_token`
- `oidc`
- `hybrid`
- `trusted_proxy`

其中：

- `headers` 用于本地开发和测试。
- `api_token` 用于系统间调用。
- `oidc` 支持 Bearer Token。
- `hybrid` 支持 API Token、OIDC 和 header 共存。
- `trusted_proxy` 适合挂在企业 SSO 网关后面。

## 11. OIDC 做到什么程度了？

**问题**

OIDC 是不是只是占位？

**回答**

不是占位，已经支持两种 Bearer Token 校验方式：

- `HS256`，通过 `OIDC_JWT_SECRET`
- `RS256 + JWKS`，通过 `OIDC_JWKS_URL`

这意味着如果企业 IdP 是标准 OIDC 提供方，可以通过 JWKS 做标准公钥校验，而不是只支持对称签名。

## 12. 为什么同时支持 API Token？

**问题**

有了 OIDC 为什么还要 API Token？

**回答**

因为企业平台会有大量机器对机器调用场景，比如：

- 外部业务系统调知识库问答
- 定时同步任务调导入接口
- 运维脚本调管理接口

这些场景用 OIDC 不一定方便，API Token 更适合服务账号和自动化接入。

## 13. API Token 是怎么设计的？

**问题**

API Token 设计上有哪些安全考虑？

**回答**

主要考虑了三点：

1. 明文 token 只在创建时返回一次。
2. 数据库只存 token hash，不存明文。
3. 支持状态管理、过期时间和最后使用时间。

这符合比较基本的企业凭证治理要求。

## 14. 为什么要记录审计日志？

**问题**

审计日志记录了什么，有什么价值？

**回答**

审计日志用于回答三个问题：

- 谁做了什么操作？
- 作用于哪个资源？
- 成功还是失败？

当前会记录：

- 租户创建
- 知识库创建和策略变更
- 文档导入
- 问答请求
- 反馈提交
- API Token 创建和吊销
- 失败请求

这对于安全审计、问题排查和合规都很重要。

## 15. 为什么要做统一错误码？

**问题**

为什么不直接返回默认的 FastAPI 错误？

**回答**

因为平台对前端、运维和调用方来说，错误需要可编程处理。

统一错误结构的价值是：

- 前端可以稳定识别错误类型。
- 网关和调用方可以按错误码做重试和提示。
- 审计里也能记录标准化结果。

## 16. 为什么还要做评测模块？

**问题**

评测模块对 RAG 平台的价值是什么？

**回答**

RAG 项目上线后最大的风险是“能回答，但不知道答得对不对”。

评测模块的作用是：

- 让问题集和预期结果结构化。
- 让不同策略的效果可对比。
- 给召回、引用命中和反馈建立质量闭环。

如果没有评测，后续优化会变成拍脑袋。

## 17. 运营指标为什么重要？

**问题**

为什么要做租户级运营指标？

**回答**

因为平台上线后，不是只有研发关心它。

运营指标能帮助回答：

- 哪个租户问答量最高？
- 失败率高不高？
- 正向反馈有没有提升？
- 评测结果有没有变差？

这些指标决定平台是否值得继续投入。

## 18. 为什么用 FastAPI + SQLAlchemy？

**问题**

技术选型为什么是这套？

**回答**

原因比较直接：

- FastAPI 适合快速构建结构化 API，类型约束和文档生成都好。
- SQLAlchemy 适合做企业项目里的 ORM 和多数据库兼容。
- Python 生态对 LLM、Embedding、文档解析和异步任务更成熟。

这套栈在“平台开发效率”和“AI 生态兼容性”之间是比较均衡的。

## 19. 为什么用 Celery？

**问题**

为什么任务系统没有自己造轮子？

**回答**

因为文档处理、索引构建、本地重试这些需求已经很典型，Celery 足够成熟。

当前支持：

- `local`，便于本地开发
- `celery`，便于真实异步链路部署

这样能兼顾开发体验和生产路径。

## 20. 如果你继续做这个项目，下一步最重要的事情是什么？

**问题**

从当前状态往前推进，优先级最高的是什么？

**回答**

我会优先做三件事：

1. 跑通真实基础设施联调  
把 PostgreSQL、Redis、Celery、`pgvector` 真实跑通。

2. 跑通真实模型网关  
用企业实际的 OpenAI-compatible 网关做 `providers/probe`、文档索引和问答链路验证。

3. 补生产级认证与接入  
把真实 OIDC/SSO、API Token、网关代理接入方式收口成标准部署方案。

因为这三件事决定项目是否真正能上线试运行。

## 21. 这个项目还有哪些明显的未完成项？

**问题**

如果你来 code review，你会说它还缺什么？

**回答**

当前主要还缺：

- 真正的文档解析能力，比如 PDF/Word/OCR
- 真实 rerank 模型接入
- 更完整的限流、配额和租户治理
- Prometheus / OpenTelemetry 等标准观测接入
- 更细粒度的资源授权
- 前端管理台
- 更完整的生产环境联调和压测

所以它现在是“非常完整的后端平台骨架”，但还不是完全收尾的生产系统。

## 22. 你觉得这个项目最核心的设计取舍是什么？

**问题**

如果只说一个设计取舍，你会强调什么？

**回答**

我会强调“平台优先，而不是能力堆砌”。

也就是先把这些事情做对：

- 契约稳定
- 数据可追踪
- 能力可替换
- 认证可治理
- 交付可运行

然后再去扩展模型、解析器和向量后端。

因为企业项目最怕的不是能力少，而是能力堆上去了但无法治理、无法运维、无法交付。

## 23. 为什么 `QueryService.search` 要先查知识库配置再做 query embedding？

**问题**

为什么不直接固定用一个 embedding model 做 query 向量化？

**回答**

因为索引时和检索时最好使用同一 embedding 模型，至少要保持兼容维度和语义空间一致。

这个项目已经支持知识库级配置：

- `embedding_model`
- `llm_model`
- `temperature`
- `max_tokens`

这样不同知识库可以针对不同文档域使用不同模型。例如：

- 法务知识库用偏精确的 embedding
- 客服知识库用偏通用的 embedding

如果 query 侧固定写死，知识库级策略就失效了。

## 24. 为什么 `QueryRecord` 里要落 `trace_id` 和 `top_k`？

**问题**

这些字段看起来像运行时参数，为什么要持久化？

**回答**

因为后续排查问答结果时，需要知道“当时是以什么配置跑出来的”。

`trace_id` 的作用：

- 把 query、retrieval log、评测结果、审计事件串起来

`top_k` 的作用：

- 判断召回深度是否影响结果
- 对比策略调优前后的差异

如果只保留最终答案，不保留过程参数，问题定位会非常困难。

## 25. 为什么 `retrieval_logs` 需要存 `content_snapshot`？

**问题**

不是已经有 `chunk_id` 和 `document_id` 了吗，为什么还存内容快照？

**回答**

因为知识库内容会变，chunk 也可能重建。

如果只存 `chunk_id`：

- 后续重建后原 chunk 可能已经不存在
- 原内容可能已被覆盖

存 `content_snapshot` 的价值是：

- 保留问答发生当下的召回证据
- 方便线上故障复盘
- 便于审计和合规取证

这本质上是在保留“检索时刻的事实快照”。

## 26. 为什么 API Token 明文只返回一次？

**问题**

如果用户丢了 token，为什么不允许再次查看？

**回答**

因为安全设计上不应该能从服务端恢复凭证明文。

服务端只存 hash，有几个好处：

- 数据库泄露时风险更低
- 管理员不能反查用户密钥
- 设计上更接近密码学最佳实践

如果用户丢失 token，正确做法是吊销旧 token、重新签发新 token，而不是把明文重新展示出来。

## 27. `hybrid` 认证模式为什么优先 API Token，再 Bearer，再 header？

**问题**

这个顺序是怎么考虑的？

**回答**

顺序反映的是“明确性”和“安全性”的优先级。

1. API Token  
最明确，通常就是系统调用，而且格式可识别。

2. Bearer Token  
通常对应 OIDC/JWT，优先级也高于开发模式 header。

3. Header 回退  
只适合开发和测试，安全性最低。

如果把 header 放前面，就可能掩盖真实认证问题，甚至让请求误走低安全路径。

## 28. 为什么 OIDC 同时支持 `HS256` 和 `RS256 + JWKS`？

**问题**

只支持标准 `RS256` 不就够了吗？

**回答**

从规范和生产环境角度，`RS256 + JWKS` 更标准；但从企业实际接入情况看，很多内部网关或早期系统仍然会给 `HS256`。

同时支持两种方式的好处是：

- 兼容不同企业现状
- 便于从旧网关平滑迁移到标准 OIDC

但生产最佳实践仍然是优先用：

- `RS256`
- `OIDC_JWKS_URL`

因为共享对称密钥的风险更高。

## 29. 为什么中间件里要自己拦认证异常并转成标准响应？

**问题**

不是已经有全局异常处理了吗？

**回答**

理论上全局异常处理能兜底，但中间件阶段抛出的异常在某些栈路径下不一定按业务异常处理器收敛得足够稳定，尤其在 Starlette/FastAPI 的 middleware 包装层里。

所以这里主动在中间件里把认证异常转成统一错误响应，有两个目的：

- 保证认证失败一定返回统一格式
- 避免请求还没进入路由层就出现不一致错误行为

这是一个偏工程稳定性的处理，而不是功能性需求。

## 30. 为什么 `ProviderService.probe` 要单独存在，而不是只看 `diagnostics`？

**问题**

诊断和探测有什么本质区别？

**回答**

`diagnostics` 是被动诊断，主要看：

- 当前启用的 provider
- 当前配置参数
- provider 自报健康状态

`probe` 是主动探测，会真实发起：

- LLM 调用
- Embedding 调用

两者差别在于：

- `diagnostics` 回答“配置看起来对不对”
- `probe` 回答“实际调用通不通”

企业联调阶段，主动探测是必须的。

## 31. 为什么 `pgvector` provider 里用了原生 SQL，而不是完全 ORM 化？

**问题**

既然用了 SQLAlchemy，为什么这里没有完全用 ORM？

**回答**

因为向量类型、`<=>` 距离计算、`CREATE EXTENSION vector` 这些操作在 ORM 抽象层里并不自然。

这里使用原生 SQL 的原因是：

- 向量相似度检索本来就偏数据库能力
- SQL 更直接、可控
- 对 `pgvector` 这种扩展特性更稳定

这类场景更适合“ORM 管常规实体，SQL 管数据库特性”。

## 32. 为什么 `PgVectorStoreProvider` 里要做 `auto_create`？

**问题**

生产环境不是应该全部靠迁移吗，为什么还要允许自动建表？

**回答**

这是为了兼顾两种场景：

1. 开发/联调  
快速起环境，降低门槛。

2. 生产  
更推荐 `AUTO_CREATE_SCHEMA=false`，用 Alembic 和正式变更流程管理结构。

也就是说，`auto_create` 是开发效率选项，不是生产治理方案。

## 33. 为什么问答链路里现在仍然先做检索，再做 rerank，再生成？

**问题**

能不能直接把所有 chunk 全喂给模型？

**回答**

不能，原因很现实：

- 上下文窗口有限
- 成本高
- 噪声太多会拉低答案质量

检索和重排的存在是为了控制：

- token 成本
- prompt 噪声
- 召回质量

RAG 的核心不是“喂更多内容”，而是“喂更相关内容”。

## 34. 为什么 `prompt_template` 放在知识库级，而不是全局级？

**问题**

全局统一 prompt 不是更简单吗？

**回答**

全局 prompt 简单，但不够灵活。

不同知识库的问答风格可能不一样：

- 法务知识库要求严格引用
- 客服知识库更强调简洁说明
- 内部制度库更强调拒答和边界控制

知识库级 prompt 的好处是：

- 便于业务隔离
- 便于逐库调优
- 不会因为改一个 prompt 影响所有场景

## 35. 为什么当前重排还是 `passthrough`，这样不是没意义吗？

**问题**

如果没有真实 reranker，现在这个设计价值在哪里？

**回答**

`passthrough` 的价值不在效果，而在架构占位。

它提前把检索链路中的“rerank 阶段”固定下来，带来的收益是：

- 上层流程稳定
- 后续替换真实 reranker 不需要改业务链路
- 测试和联调可以先跑通整个流程

这是一种典型的平台化做法：先把扩展点留好，再逐步替换真实能力。

## 36. 为什么 `providers/probe` 只探测 LLM 和 Embedding，没有探测全部 provider？

**问题**

不是还有 parser、vector store、reranker 吗？

**回答**

因为从“模型网关联调”这个任务来看，最关键、最容易失败的就是：

- chat 生成
- embedding 调用

而 parser、vector store、reranker 更适合通过：

- `diagnostics`
- 索引任务
- 实际问答链路

来验证。主动探测先聚焦最不稳定、最外部化的依赖，范围更合理。

## 37. 如果第三方模型网关的 `/embeddings` 接口维度变化了，会发生什么？

**问题**

这个系统怎么发现和处理向量维度不匹配？

**回答**

如果 embedding 维度和 `VECTOR_DIMENSION` 或向量库表结构不一致，最直接的问题是：

- 向量写入失败
- 检索失败
- 或者结果不可靠

当前系统里可以通过：

- `providers/probe` 查看 embedding 维度
- `pgvector` 写入失败日志
- 索引任务失败状态

来发现问题。

更进一步的生产优化应该是：

- 在索引前主动校验 embedding 维度
- 在 provider 初始化阶段做一次维度自检

## 38. 为什么这个项目里同时有 `audit_events`、`query_records`、`retrieval_logs`，不会重复吗？

**问题**

这三类表是不是有功能重叠？

**回答**

它们关注的是不同层次：

- `audit_events`：谁做了什么操作，偏治理和合规
- `query_records`：一次问答请求本身，偏业务记录
- `retrieval_logs`：一次问答过程中的召回详情，偏调试和质量分析

看起来都和“问答”有关，但职责完全不同。如果混在一张表里，后面会越来越难维护。

## 39. 为什么 Repository 这一层是有价值的？

**问题**

很多人会说 SQLAlchemy 已经是抽象了，为什么还要再包一层 repository？

**回答**

Repository 的价值不是“再抽象一次数据库”，而是：

- 固定数据访问边界
- 把查询逻辑从 service 里拆出来
- 便于替换底层实现
- 便于测试和复用

对于平台型项目，这一层会随着复杂度上升越来越有价值。
